Spec-driven workflow:
1. Define the problem in one sentence.
2. List users, success criteria, non-goals, and constraints.
3. Describe the proposed architecture and affected files.
4. Define data flow, API contracts, UI states, error handling, and security boundaries.
5. Define test strategy before implementation.
6. Identify assumptions and open questions.
7. Pressure-test the spec before coding.
8. Save the spec under `docs/specs/YYYY-MM-DD-<topic>.md` when the change is substantial.

Do not implement until the spec is clear and approved.
